Features

Built for terminal natives.

One command. The GIF plays. gifterm exits. The animation lives on.

▒

GPU-native

Animations persist in kitty's GPU memory after gifterm exits. Like an <img> tag for your terminal. Clear the screen to dismiss.

⇄

Content-addressed cache

Frames are decoded once, cached by SHA-256 hash. Second play loads instantly from disk, no decode step. Width variants get their own cache key.

▸

Fire and forget

gifterm exits immediately after transmitting frames. The animation loops on the GPU independently. Multiple GIFs can run simultaneously.

⇢

Smart scaling

Pass --width 400 and gifterm scales with Lanczos3 filtering. Original aspect ratio preserved. Scaled frames are cached separately.

⚙

Zero runtime deps

Single static Rust binary. No Python, no Node, no runtime dependencies. Compiles to one file you can drop anywhere.

★

Library + CLI

Use as a CLI tool or import as a Rust crate. The library compiles without the CLI feature, no clap dependency. Works with any kitty-protocol integration.

How it works

Three steps, then gifterm gets out of the way.

Decode

GIF frames are decoded into raw RGBA buffers using the image crate. Optionally scaled down with Lanczos3 filtering for crisp results at any width.

Cache

Decoded frames are written to a content-addressed cache at ~/.cache/gifterm/, keyed by SHA-256 of the source file. Next time, decode is skipped entirely.

Transmit

Frames are sent to the terminal via the kitty graphics protocol using temp-file transfer. The terminal assembles them into a looping animation on the GPU.

// Library usage
use std::path::Path;

let path = Path::new("animation.gif");
let (meta, frames) = gifterm::load_frames(path, Some(400))?;
gifterm::play(&meta, &frames)?;
